# 02. 预防性维护与风险发现

## 1. 模块目标

预防性维护不是让用户“填完点检表”，而是提前发现风险，并把风险推进到处理闭环。

核心主线：

```mermaid
flowchart LR
    A[设备类型/等级] --> B[维护基准]
    B --> C[设备级计划]
    C --> D[自动生成任务]
    D --> E[扫码执行]
    E --> F[异常识别]
    F --> G[异常事件/维修工单]
    E --> H[设备履历]
```

## 2. 业务类型

| 类型 | 目的 | 特点 |
|------|------|------|
| 点检 | 高频、轻量、关键点确认 | 通常班次/每日执行 |
| 巡检 | 区域或路线检查 | 可按路线、位置批量执行 |
| 保养 | 周期性维护和更换 | 项目更重，可要求验收和备件 |

产品模型建议统一为“维护任务”，通过业务类型区分点检、巡检、保养。可选方案如下：

| 方案 | 说明 | 推荐度 |
|------|------|--------|
| A. 点检、巡检、保养完全独立 | 三套计划、三套任务、三套执行页面 | 不推荐，重复建设多 |
| B. 统一维护任务，业务类型区分 | 共用计划、任务、项目、异常、履历模型 | 推荐，通用且可扩展 |
| C. 点检/巡检合并，保养独立 | 检查类统一，保养保留独立模型 | 可选，适合保养流程很重的企业 |

推荐：B。原因是三类业务都遵循“基准 → 计划 → 任务 → 执行 → 异常/履历”的结构，统一底层最利于扩展。

## 3. 基准与计划

| 对象 | 说明 |
|------|------|
| 类型基准 | 按设备类型配置默认项目和频次 |
| 等级策略 | A/B/C 或关键/普通设备使用不同频率和异常策略 |
| 设备级基准 | 单台设备继承类型基准后允许微调 |
| 维护计划 | 根据设备级基准生成周期任务 |
| 维护任务 | 最终派给人员执行的单据 |

## 4. 检查项目字段

| 字段 | 说明 |
|------|------|
| 项目名称 | 如温度、震动、油位、异响 |
| 结果类型 | 正常/异常、数值、文本、图片、选择项 |
| 单位 | 数值型项目使用 |
| 上下限 | 阈值判断使用 |
| 是否必填 | 必检项必须填写 |
| 是否关键项 | 关键项异常可升级 |
| 异常动作 | 仅记录、生成异常事件、建议转维修 |

## 5. 异常识别

| 规则 | 示例 | MVP 建议 |
|------|------|----------|
| 阈值规则 | 温度大于 80℃ | 必做 |
| 必填缺失 | 关键项未填 | 必做 |
| 连续异常 | 7 天内异常 3 次 | 增强 |
| 趋势异常 | 震动连续上升 | 增强 |
| 同类对比 | 同型号设备偏高 | 可选 |
| AI 判断 | 图片/文本判断风险 | 可选 |

## 6. 任务执行交互

推荐移动端优先：

1. 扫设备二维码。
2. 展示当前待执行任务。
3. 进入任务后只展示本次需要执行的项目。
4. 支持拍照、录入数值、选择异常。
5. 异常项要求填写说明或图片。
6. 提交后生成履历，必要时生成异常事件。

## 7. 待确认事项

### 7.1 点检和巡检路线管理

| 方案 | 说明 | 优点 | 风险 |
|------|------|------|------|
| A. 不做路线 | 任务只按设备生成 | 简单 | 区域巡检体验不足 |
| B. 巡检支持路线，点检不强制 | 巡检可按路线批量执行，点检按设备执行 | 符合现场习惯 | 多一套路线路配置 |
| C. 点检、巡检都强制路线 | 所有任务必须挂路线 | 管理严谨 | 小企业使用负担重 |

推荐：B. 巡检支持路线，点检不强制。

推荐原因：巡检通常按区域/路线走，点检通常围绕单台或岗位设备。分开处理更合理。

### 7.2 保养验收策略

| 方案 | 说明 | 优点 | 风险 |
|------|------|------|------|
| A. 全部验收 | 所有保养完工后验收 | 质量强管控 | 流程重 |
| B. 全部免验收 | 提交即完成 | 操作快 | 关键保养不可控 |
| C. 按设备等级/保养等级配置 | 关键设备和重要保养必须验收 | 平衡效率和质量 | 需要配置规则 |

推荐：C. 按设备等级/保养等级配置。

推荐原因：普通清洁、润滑不必都验收，关键设备大保养必须可控。

### 7.3 节假日、停产、检修窗口跳过规则

| 方案 | 说明 | 优点 | 风险 |
|------|------|------|------|
| A. 不支持跳过 | 到期就生成任务 | 简单 | 停产期间产生大量无效逾期 |
| B. 支持日历跳过 | 节假日、停产日不生成或顺延 | 符合多数企业 | 需要维护生产日历 |
| C. 支持复杂窗口策略 | 可按产线、设备、班次设置窗口 | 精细 | 首版复杂 |

推荐：B. 支持日历跳过。

推荐原因：维护计划必须考虑非工作日和停产。首版用生产日历顺延即可，不先做复杂班次窗口。

### 7.4 异常自动转维修

| 方案 | 说明 | 优点 | 风险 |
|------|------|------|------|
| A. 全部自动转维修 | 任一异常都生成维修工单 | 不漏处理 | 工单噪音大 |
| B. 全部人工确认 | 异常先进入事件池，由责任人判断是否转单 | 控制准确 | 依赖人工及时处理 |
| C. 严重异常自动转，普通异常人工确认 | 按异常等级、关键项、设备等级配置 | 兼顾效率和准确 | 需要规则配置 |

推荐：C. 严重异常自动转，普通异常人工确认。

推荐原因：关键设备和高风险异常不能等人工筛选，普通异常直接转维修会造成工单膨胀。
