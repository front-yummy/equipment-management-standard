# 05. 运行状态与OEE指标

## 1. 模块目标

运行状态与指标模块负责回答：

1. 设备现在是否在生产。
2. 停机损失来自哪里。
3. 维修效率和设备可靠性如何。
4. 指标异常后应该下钻到什么业务动作。

## 2. E10 状态

| 状态 | 含义 |
|------|------|
| PT | 正常生产 |
| UD | 故障停机 |
| SD | 计划停机 |
| SB | 待机等待 |
| EN | 工程试验 |
| NS | 无生产计划 |

## 3. 指标设计

| 指标 | 数据来源 | 首版建议 |
|------|----------|----------|
| MTTR | 维修工单节点时间 | MVP |
| MTBF | 故障次数 + 运行时间 | 增强 |
| 停机时长 | 停机记录/维修工单 | MVP 可轻量 |
| OEE | 生产计划、运行、产出、良率 | 增强 |
| 保养达成率 | 保养计划和任务 | MVP |
| 异常闭环率 | 异常事件 | MVP |

## 4. 指标闭环

```mermaid
flowchart LR
    A[指标异常] --> B[下钻定位]
    B --> C[设备/位置/班组/原因]
    C --> D[发起动作]
    D --> E[维修/保养/备件/改善]
    E --> F[结果回写]
    F --> A
```

## 5. 待确认事项

### 5.1 OEE 是否首版必做

| 方案 | 说明 | 优点 | 风险 |
|------|------|------|------|
| A. 首版完整 OEE | 时间稼动率、性能稼动率、良率全部覆盖 | 管理价值完整 | 依赖生产计划、产出、良率，首版复杂 |
| B. 首版只做停机和维修指标 | 先做停机、故障、MTTR、保养达成率 | 落地快 | 缺少完整效率评价 |
| C. 首版做 OEE 数据底座，不做完整看板 | 先采集 E10、停机损失、标准节拍字段 | 为后续 OEE 铺路 | 首版展示价值有限 |

推荐：B + C。首版做停机和维修指标，同时预留 OEE 数据底座。

推荐原因：先让设备侧数据跑起来，避免一开始被生产、质量、MES 集成拖住。

### 5.2 运行状态来源

| 方案 | 说明 | 优点 | 风险 |
|------|------|------|------|
| A. 数采自动获取 | PLC/SCADA/IoT 自动上报状态 | 实时准确 | 接入成本高 |
| B. MES 获取 | 从 MES 取设备状态或生产状态 | 和生产计划一致 | MES 状态不一定等同设备状态 |
| C. 人工填报 | 人员手工维护停机和状态 | 上线快 | 准确性依赖人 |
| D. 多来源融合 | 数采优先，MES 辅助，人工可修正并留痕 | 最通用 | 规则设计复杂 |

推荐：D. 多来源融合。

推荐原因：不同客户数据基础差异大。标准产品应允许从人工起步，逐步接入数采和 MES。

### 5.3 MTTR 默认起点

| 方案 | 说明 | 优点 | 风险 |
|------|------|------|------|
| A. 报修时间 | 从工单创建开始算 | 反映完整停机恢复体验 | 包含派工等待，维修人员可能不认可 |
| B. 派工时间 | 从主管派工开始算 | 反映维修组织响应 | 忽略报修到派工等待 |
| C. 接单时间 | 从维修人员接单开始算 | 更聚焦维修执行效率 | 管理层看不到派工慢 |
| D. 可配置起点，同时展示拆分耗时 | 默认接单时间，但保留报修、派工、到场、维修拆分 | 兼顾管理和执行 | 指标说明要清楚 |

推荐：D. 可配置起点，同时展示拆分耗时。

推荐原因：不同企业 MTTR 口径不同。默认用接单时间更公平，但必须展示报修到派工、派工到接单等分段耗时。
