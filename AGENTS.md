# AGENTS.md

## 项目说明

本仓库用于沉淀设备管理系统的产品需求、工程化需求分析和后续实现资料。当前核心文档为：

- `设备管理系统PRD.md`：产品总览、范围、模块索引。
- `需求分析/`：按模块拆分的工程化需求目录，服务页面绘制、代码实现、测试验收和需求迭代。
- `原型/`：墨刀原型导出资料。
- `参考/`：需求参考来源。

## 协作原则

1. 修改需求前，先确认涉及的模块文件，不要把所有细节继续堆回总 PRD。
2. 总 PRD 只维护业务全景、范围边界、模块索引和跨模块摘要。
3. 深层逻辑写入 `需求分析/` 下对应模块，包括触发条件、状态流转、边界条件、数据回写、集成规则和验收口径。
4. 原型、详细设计、操作手册和设备需求文档出现冲突时，不直接覆盖结论；先在模块文件的“待澄清与迭代事项”记录差异。
5. 涉及状态机、定时任务、跨系统回写、指标口径的改动，必须同步更新验收口径。
6. 本项目目标是通用标准产品文档，不是具体客户项目复刻；遇到复杂、冷门、强现场绑定需求时，优先抽象为可配置项、可选扩展或待澄清事项。
7. 文档应优先保证 MVP 主流程简单好用，再描述增强能力；不要让低频复杂流程阻塞基础产品落地。

## 写作规范

1. 使用中文维护需求文档。
2. 优先使用表格描述页面、字段、状态和验收口径。
3. 流程复杂时使用 Mermaid 图。
4. 每条规则尽量写成可验证的句子，避免“支持灵活配置”“优化体验”等空泛表述。
5. 对暂未确认的内容，写入“待澄清与迭代事项”，不要伪装成已确认需求。

## 后续任务建议

1. 页面设计前：从模块文件提取页面清单、字段、操作按钮、状态可见性。
2. 接口设计前：从模块文件提取状态机、触发条件、数据来源、回写目标。
3. 测试设计前：从模块文件提取主流程、异常分支、边界条件和验收口径。
4. 需求变更时：先更新模块文件，再视影响同步总 PRD 索引。
