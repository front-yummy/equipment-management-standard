# 设备管理系统需求分析目录

本目录由 `设备管理系统PRD.md` 拆分而来，并补充分析了原型、详细设计文档、EAM 项目用户操作手册、设备需求文档中的深层业务逻辑。目录目标是让需求可以直接服务页面绘制、接口设计、代码实现、测试验收和后续迭代。

## 产品化取舍原则

本目录面向通用标准产品，而不是复刻某个具体项目。参考文档中的复杂、冷门、强客户绑定需求，按以下规则处理：

1. **主流程简单好用**：优先沉淀设备台账、点巡检/保养、维修工单、OEE、备件、报表这些通用闭环。
2. **项目特性参数化**：例如通知渠道、外部系统名称、采集阈值、编辑时间窗、审批系统，默认写成可配置能力。
3. **复杂能力插件化**：例如高级 AI 自动制单、九宫格分类、复杂库存风控、多级审批回写，作为可选扩展，不阻塞基础版本。
4. **冷门规则不硬编码**：来自原型备注或单一客户现场的规则，进入“推荐默认值”或“待澄清与迭代事项”。
5. **落地优先于完整**：若一个需求实现成本高但通用价值低，文档应给出简化版 MVP 和后续增强路径。
6. **先整体后具体**：模块文档先讲目标、页面、流程、状态和规则，再在靠后的“页面字段清单”中细化字段。
7. **先录入后展示**：字段清单内部优先描述新增/编辑/执行表单，再描述详情、列表和统计字段。

## 模块文件统一结构

每个模块文件按以下结构维护：

1. 模块目标与边界
2. 角色与权限
3. 页面清单
4. 主业务流程
5. 状态机与触发条件
6. 字段与数据规则或业务规则
7. 页面字段清单
8. 跨模块/外部系统联动
9. 异常分支与边界条件
10. 验收口径
11. 待澄清与迭代事项

## 工程化使用方式

页面绘制时，先看“页面清单、主业务流程、状态机”，再看靠后的“页面字段清单、操作按钮、状态可见性”。

代码实现时，以“状态机、触发条件、跨模块联动、异常分支”为准。

测试验收时，以“主流程、异常分支、数据回写、统计口径”为准。

需求迭代时，先在对应模块的“待澄清与迭代事项”记录，再在评审确认后固化到正式规则中。
