# 设备管理系统需求分析目录

本目录由 `设备管理系统PRD.md` 拆分而来，并补充分析了原型、详细设计文档、EAM 项目用户操作手册、设备需求文档中的深层业务逻辑。目录目标是让需求可以直接服务页面绘制、接口设计、代码实现、测试验收和后续迭代。

## 产品化取舍原则

本目录面向通用标准产品，而不是复刻某个具体项目。参考文档中的复杂、冷门、强客户绑定需求，按以下规则处理：

1. **主流程简单好用**：优先沉淀设备台账、预防性维护、维修工单、OEE、备件、仓库和主数据这些通用闭环。
2. **领域边界清晰**：主数据、仓库、工作流、消息、AI、系统管理作为独立能力；设备业务模块只声明依赖和联动。
3. **可独立部署**：`base + 设备核心模块` 必须可独立部署；备件管理如启用，必须同时启用轻量 warehouse 或对接 WMS。
4. **录入必有导入**：凡是列表型维护、配置型维护、主数据维护页面，都必须支持模板下载、导入、导出和错误报告。
5. **外部依赖可降级**：每个外部系统依赖都必须提供轻量本地闭环或明确为可选扩展；无外部系统时支持人工导入、手动回写或本地状态维护。
6. **项目特性参数化**：例如通知渠道、外部系统名称、采集阈值、编辑时间窗、审批流程，默认写成可配置能力。
7. **复杂能力插件化**：例如高级 AI 自动制单、九宫格分类、复杂库存风控、多级审批回写，作为可选扩展，不阻塞基础版本。
8. **冷门规则不硬编码**：来自原型备注或单一客户现场的规则，进入“推荐默认值”或“待澄清与迭代事项”。
9. **落地优先于完整**：若一个需求实现成本高但通用价值低，文档应给出简化版 MVP 和后续增强路径。
10. **先整体后具体**：模块文档先讲目标、页面、流程、状态和规则，再在靠后的“页面字段清单”中细化字段。
11. **先录入后展示**：字段清单内部优先描述新增/编辑/执行表单，再描述详情、列表和统计字段。

## 模块文件统一结构

每个模块文件按以下结构维护：

1. 模块目标与边界
2. 角色与权限
3. 页面清单
4. 主业务流程
5. 状态机与触发条件
6. 字段与数据规则或业务规则
7. 页面字段清单
8. 跨模块/外部系统联动
9. 异常分支与边界条件
10. 验收口径
11. 待澄清与迭代事项
