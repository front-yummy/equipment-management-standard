# 04. AI 知识库与智能助手

## 模块目标与边界

AI 知识库是独立模块，服务设备模块的维修、OEE、备件和预防性维护场景。它覆盖故障知识库、维修推荐、ChatAI 问答、备件领用助手和指标分析。

标准产品应先保证知识库与工单推荐可落地，自动制单和深度指标分析作为增强能力；EAM 负责提供结构化知识、业务上下文和调用入口，外部 AI 服务负责向量化、检索和生成。

## 页面与入口

| 入口 | 场景 |
|------|------|
| 知识库管理 | 人工录入、模板导入、版本维护、AI 同步 |
| 维修工单 AI 推荐 | 根据设备和故障描述推荐原因与措施 |
| ChatAI - 知识库 | 维修知识问答、SOP 查询 |
| ChatAI - 备件领用申请 | 可选扩展：查询库存、生成出库单草稿 |
| ChatAI - 指标分析 | 可选扩展：分析 KPI 异常、趋势、突变和关联工单 |


## 知识库维护流程

1. 知识库管理员新增知识条目。
2. 填写标题、适用设备类型、设备名称、版本号、关键词标签。
3. 添加 SOP、故障排除列表等模块内容。
4. 故障排除列表字段包括设备状态、故障代码、原因、类型、描述、举措、关联部件、MTTR、预防性建议。
5. 保存后状态为“未同步”。
6. 用户手动触发 AI 同步。
7. 同步成功后状态为“已同步”；失败则保持原状态并记录日志。
8. 条目编辑或版本更新后，同步状态重置为“未同步”。

## 维修推荐流程

```mermaid
flowchart LR
    A[执行中维修工单] --> B[点击 AI 推荐]
    B --> C[组装上下文: 设备类型/设备/停机详情/故障描述]
    C --> D[检索知识库和历史工单]
    D --> E[返回多个方案]
    E --> F[用户采纳/判定有效无效]
    F --> G[写入原因和措施字段]
    G --> H[用户编辑并完工]
    H --> I[闭环案例沉淀]
```

推荐规则：

1. 输出格式固定为“故障原因 + 解决方案”。
2. 支持多方案展示，例如方案 1、方案 2、方案 3。
3. 用户可标记有效、无效或重新生成。
4. 事件描述变更后再次点击 AI，重新生成方案。
5. 事件描述未变化时，再次点击可仅控制展示隐藏。
6. AI 生成内容必须允许人工编辑，不能直接锁定。

## ChatAI 备件领用流程

1. 用户选择“备件领用申请”场景。
2. 用户输入备件型号、领取数量、厂家、领取人员等信息。
3. AI 查询备件库存，以表格返回库存位置和数量。
4. 用户确认后，AI 生成出库单草稿。
5. 单据内容包含领用人、领用部门、领用原因和备件详情。
6. 建议最终提交前仍需人工确认。

## ChatAI 指标分析流程

1. 用户在 KPI 看板或 ChatAI 中选择指标分析场景。
2. 系统提供筛选条件、指标值、趋势、同环比、突变点、关联工单。
3. AI 生成异常点解读、关联因素、故障原因总结和改进建议。
4. 输出仅作为决策辅助，不改变底层指标结果。

## 数据来源

| 数据 | 来源 |
|------|------|
| 设备文档、SOP、手册 | 知识库上传 |
| 历史维修工单 | 已完成异常工单 |
| 故障原因和措施 | 维修工单闭环字段 |
| 备件库存 | 备件台账/库存系统同步数据 |
| KPI 指标 | OEE/KPI 预计算或实时查询 |

## 跨模块联动

1. 维修工单提供设备、故障描述、报警详情、历史工单和处理结果，AI 返回原因与措施建议。
2. OEE 提供指标、趋势、停机损失和关联工单，AI 返回异常解读和改善建议。
3. 备件管理提供备件需求上下文，warehouse 提供库存查询结果，AI 可生成领用草稿。
4. 预防性维护可引用知识库中的预防性建议，用于后续优化点巡检/保养基准。
5. 系统管理提供用户和权限范围，AI 查询和生成结果必须受数据权限控制。

## 验收口径

1. 知识条目变更后，同步状态必须变为未同步。
2. AI 推荐结果采纳后写入工单字段，且用户可编辑。
3. AI 推荐需记录采纳、无效、重新生成等反馈。
4. ChatAI 查询库存时，结果与备件台账库存一致。
5. 指标分析能引用筛选范围，不跨越用户权限数据范围。
6. 知识条目支持模板下载、导入、导出；导入失败可下载错误报告。

## 待澄清与迭代事项

1. AI 服务接口协议、鉴权、超时、失败降级需补充技术细节。
2. 历史工单自动沉淀到知识库是否需要人工审核。
3. AI 生成出库单是草稿还是可直接提交审批，需要业务确认。
